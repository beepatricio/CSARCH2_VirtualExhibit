  In computer architecture, a single machine instruction is rarely executed as one solitary action. ## Fetch Stage
<div style={{ textAlign: "justify" }}>
The Fetch stage is the first step of the <span className="font-mono text-stone-200 text-[11px] bg-stone-900 px-1.5 py-0.5 border border-stone-850 rounded inline-block mx-1">
Fetch-Decode-Execute</span> cycle. Its primary purpose is to retrieve the next instruction from <span className="font-mono text-stone-200 text-[11px] bg-stone-900 px-1.5 py-0.5 border border-stone-850 rounded inline-block mx-1">Memory</span> 
and safely load it into the <span className="font-mono text-stone-200 text-[11px] bg-stone-900 px-1.5 py-0.5 border border-stone-850 rounded inline-block mx-1">IR</span> (Instruction Register) through a sequence of micro-operations. 
The process begins with the <span className="font-mono text-stone-200 text-[11px] bg-stone-900 px-1.5 py-0.5 border border-stone-850 rounded inline-block mx-1">PC</span>, which stores the address of the next instruction. 
This address is transferred to the <span className="font-mono text-stone-200 text-[11px] bg-stone-900 px-1.5 py-0.5 border border-stone-850 rounded inline-block mx-1">MAR</span>, allowing memory to locate the instruction. 
The retrieved instruction is temporarily stored in the <span className="font-mono text-stone-200 text-[11px] bg-stone-900 px-1.5 py-0.5 border border-stone-850 rounded inline-block mx-1">MBR</span> 
before finally being loaded into the <span className="font-mono text-stone-200 text-[11px] bg-stone-900 px-1.5 py-0.5 border border-stone-850 rounded inline-block mx-1">IR</span>, where it is prepared for the Decode stage.
</div>

<h3 class="text-xl font-bold mt-6 mb-4">Key Components of the Fetch Stage</h3>
* **Program Counter (PC):** Holds the memory address of the next instruction to be fetched. The CPU uses the value stored in the PC to determine which instruction should be retrieved from memory.
* **Memory Address Register (MAR):** Stores the memory address received from the Program Counter. This address is then used to access the correct location in memory.
* **Memory:** Stores program instructions and data. During the Fetch stage, memory uses the address in the MAR to locate and retrieve the requested instruction.
* **Memory Buffer Register (MBR):** Temporarily holds the instruction retrieved from memory before it is transferred to the Instruction Register.
* **Instruction Register (IR):** Stores the fetched instruction once it has been transferred from the MBR. The instruction remains in the IR until the CPU begins the Decode stage.

## Decode Stage
Once the Fetch stage finishes, the instruction in the {" "} <span className="font-mono text-stone-200 text-[11px] bg-stone-900 px-1.5 py-0.5 border border-stone-850 rounded inline-block mx-1">IR</span> 
(Instruction Register) is ready to be decoded. These are raw bits that are unrecognizable unless interpreted and split
into recognizable parts. That splitting and interpretation is the core of the decode stage.

<h3 class="text-xl font-bold mt-6 mb-4">Core Component Breakdown</h3>

* **Opcode:** Part of the instruction that tells what operation to perform. Short for 'operation code', it tells the CPU which action to take (e.g., ADD, MOV, SUB, or JMP). This is read first to figure out everything else.
* **Operand(s):** Values or addresses that the instruction operates on. These are the data items that the instruction uses or modifies. For example, in the instruction *'ADD R1, R2'*, the operands are R1 and R2.
* **Control Unit:** The part of the CPU that directs the flow of data and operations. Once the opcode is known, the Control Unit generates control signals that tell the rest of the CPU how to execute the instruction, such as which data paths to use.
* **Instruction Decoder:** A hardware component that takes the instruction from the Instruction Register (IR) and splits it into its opcode and operand fields.

><h3 class="text-xl font-bold mt-6 mb-4"> Operational Takeaways</h3>

> The Decode stage splits every incoming instruction into two primary components:
> 1. A separated **opcode** + **operand(s)** pair.
> 2. A route from the Control Unit to tell the Execute stage what to handle next.

## Execute Stage
<div style={{ textAlign: "justify" }}>
After the opcode and operands are identified in the Decode stage, the instruction's effect gets carried out. Unlike Fetch and Decode, which always follow the same fixed sequence, Execute has no single path. The Control Unit uses the decoded opcode to select one of several routes, of which are a register-to-register move through the ALU, a memory read or write through the MAR and MBR, an immediate value straight from the IR, or a branch that overwrites the PC. It then activates the control signals needed to carry data along it. Once execution finishes, the cycle restarts from Fetch to process the next instruction in memory.
</div>

<h3 class="text-xl font-bold mt-6 mb-4">Key Components of the Execute Stage</h3>

* **Arithmetic Logic Unit (ALU):** Performs arithmetic (ADD, SUB) and logical operations. Even a plain register-to-register move passes through the ALU, since it sits on the CPU's main internal data path.
* **Temp Register:** A small holding register that briefly stages a value on its way into the ALU, keeping the ALU's inputs and outputs from overlapping mid-operation.
* **General-Purpose Registers:** Hold the operand values an instruction reads from, and store the result it writes back to.
* **Memory Address Register (MAR) & Memory Buffer Register (MBR):** As reused from the Fetch stage, the MAR holds the target address for a memory access, and the MBR temporarily holds the value being read from or written to memory.
* **Program Counter (PC):** Normally just holds the next address to fetch, but for branch/jump instructions, Execute overwrites it directly to redirect the CPU.

> <h3 class="text-xl font-bold mt-6 mb-4"> Operational Takeaways</h3>
>
> The Execute stage is where the Fetch-Decode-Execute cycle closes the loop:
> 1. The route chosen by the Control Unit determines which components activate as not every instruction touches the same hardware.
> 2. Whether the result lands in a register, in memory, or in the Program Counter itself, execution ends with the CPU ready to fetch its next instruction.
